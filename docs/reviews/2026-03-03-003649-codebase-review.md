## CRITICAL

### BUG-1: Cycle detection misses waits-for dependencies

- **File:** `src/storage/sqlite.rs:448`
- **Category:** Logic bug
- **Severity:** Critical

`check_cycle` with `blocking_only=true` filters to `('blocks', 'parent-child', 'conditional-blocks')` but omits `'waits-for'`, despite `DependencyType::is_blocking()` and `rebuild_blocked_cache_impl` both treating `waits-for` as blocking.

A `waits-for` cycle (A waits-for B, B waits-for A) will not be detected and will be inserted, corrupting the dependency graph. The blocked cache will then loop indefinitely or produce wrong results when the graph is traversed.

**Evidence:**

The model definition (`src/model/mod.rs` line 254-258):

```rust
pub const fn is_blocking(&self) -> bool {
    matches!(
        self,
        Self::Blocks | Self::ParentChild | Self::ConditionalBlocks | Self::WaitsFor
    )
}
```

The blocked cache builder (`sqlite.rs` line 1494) also includes it:

```sql
WHERE d.type IN ('blocks', 'conditional-blocks', 'waits-for')
```

But `check_cycle` (line 448) omits it:

```rust
let type_filter = if blocking_only {
    "AND type IN ('blocks', 'parent-child', 'conditional-blocks')"
} else {
    ""
};
```

**Fix:** Add `'waits-for'` to the type filter string on line 448:

```rust
"AND type IN ('blocks', 'parent-child', 'conditional-blocks', 'waits-for')"
```

### BUG-2: add\_dependency cycle check is outside the transaction (TOCTOU)

- **File:** `src/storage/sqlite.rs:1965-1975`
- **Category:** Race condition
- **Severity:** Critical

The cycle guard runs before `self.mutate(...)`, so it executes outside the `BEGIN IMMEDIATE` transaction. Two concurrent agents can simultaneously pass the check and both insert their dependency, creating a cycle.

```rust
pub fn add_dependency(...) -> Result<bool> {
    // Cycle check runs HERE - no write lock held
    if let Ok(dt) = dep_type.parse::<DependencyType>()
        && dt.is_blocking()
        && self.would_create_cycle(issue_id, depends_on_id, true)?
    {
        return Err(BeadsError::DependencyCycle { ... });
    }

    // Actual write happens HERE under BEGIN IMMEDIATE
    self.mutate("add_dependency", actor, |conn, ctx| { ... })
}
```

In a concurrent multi-agent environment (the stated use case), two agents can simultaneously call `add_dependency("A","B","blocks")` and `add_dependency("B","A","blocks")`. Both read a cycle-free graph, both pass the check, and both insert their dependency, creating an A to B to A cycle.

**Compare:** `update_issue`'s claim guard correctly re-checks the assignee inside the `mutate` closure (line 498-523) for this exact reason.

**Fix:** Move the cycle check inside the `mutate` closure so it executes under the exclusive write lock.

## IMPORTANT

### BUG-3: set\_config/set\_metadata non-atomic DELETE+INSERT

- **File:** `src/storage/sqlite.rs:2964-2976, 3358-3368`
- **Category:** Data integrity
- **Severity:** Important

Both perform DELETE then INSERT on the raw connection with no wrapping transaction. If INSERT fails (disk full, I/O error, process killed), the key is silently deleted with no recovery. Same pattern in `set_export_hash` and `set_export_hashes`.

```rust
pub fn set_config(&mut self, key: &str, value: &str) -> Result<()> {
    // Step 1: DELETE (committed immediately in autocommit mode)
    self.conn.execute_with_params(
        "DELETE FROM config WHERE key = ?",
        &[SqliteValue::from(key)],
    )?;
    // Step 2: INSERT - if this fails, the key is permanently gone
    self.conn.execute_with_params(
        "INSERT INTO config (key, value) VALUES (?, ?)",
        &[SqliteValue::from(key), SqliteValue::from(value)],
    )?;
    Ok(())
}
```

Note: `set_metadata_in_tx` (line 3520) exists as a transactional version and is used from import paths. But the public `set_config` and `set_metadata` methods are not transactional.

**Fix:** Wrap in an explicit transaction or route through `self.mutate(...)`.

### BUG-4: rebuild\_blocked\_cache uses plain BEGIN instead of BEGIN IMMEDIATE

- **File:** `src/storage/sqlite.rs:1466`
- **Category:** Concurrency
- **Severity:** Important

All other write paths use `self.mutate(...)` with `BEGIN IMMEDIATE` + 5-retry exponential backoff. This path uses plain `BEGIN`, creating a deferred transaction that can fail with `SQLITE_BUSY` when it tries to acquire the write lock for `DELETE FROM blocked_issues_cache`.

```rust
pub fn rebuild_blocked_cache(&mut self, force_rebuild: bool) -> Result<usize> {
    if !force_rebuild {
        return Ok(0);
    }
    self.conn.execute("BEGIN")?;  // plain BEGIN, not BEGIN IMMEDIATE
    match Self::rebuild_blocked_cache_impl(&self.conn) { ... }
}
```

**Fix:** Use `self.mutate(...)` or `BEGIN IMMEDIATE` with retry logic.

### PERF-1: O(n^2) visited-set check in collect\_descendant\_ids

- **File:** `src/storage/sqlite.rs:2716`
- **Category:** Performance
- **Severity:** Important

BFS uses `Vec::contains()` as the visited check (O(n) per lookup). Called on every `--parent --recursive` query. `HashSet` is already imported at the top of the file.

```rust
if !result.contains(&id) {  // O(n) linear scan
    queue.push_back(id.clone());
    result.push(id);
}
```

**Fix:** Replace with a `HashSet` for O(1) membership test:

```rust
let mut visited = HashSet::new();
// ...
if visited.insert(id.clone()) {  // O(1) insert + membership test
    queue.push_back(id.clone());
    result.push(id);
}
```
